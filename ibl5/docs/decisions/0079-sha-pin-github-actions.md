# ADR-0079: SHA-pin all external GitHub Actions, enforced by a drift-guard

**Status:** Accepted
**Date:** 2026-07-05

## Context

Floating action tags (`owner/repo@v7`) are a supply-chain risk: a tag is a mutable pointer, so a moved or compromised tag silently injects new code into CI on the next run, with full access to the workflow's secrets and token. GitHub's own hardening guidance recommends pinning third-party (and, defensively, first-party) actions to a full commit SHA. This repo is especially suited to pinning because ADR-0017 already forces full CI on Dependabot PRs and auto-squash-merges them on green — so the usual objection to SHA-pinning ("SHA bumps are noisy manual chores") does not apply here: bumps arrive as low-friction, auto-merged Dependabot PRs. Prior to this change, zero external refs were SHA-pinned; all 19 distinct external actions across 26 workflows and 5 composite actions were tag-pinned.

## Decision

Pin **all** external actions to a 40-char commit SHA, generated by **pinact** (`suzuki-shunsuke/pinact`), which resolves each pin from the upstream tag and writes a trailing `# vX` version comment. Scope includes first-party `actions/*` and `github/codeql-action/*` (pinact's default; defense in depth, no first-vs-third-party carve-out). Local `./.github/actions/*` composite refs are exempt — they version with the repo and cannot be SHA-pinned. A **`pinact-check` drift-guard job** runs `pinact run --check` on every PR and is wired into the `gate` aggregator's `needs:` array (a required check via `gate`, not a new branch-protection context) — any floating tag reds the gate. Mechanically enforced by the `pinact-check` CI job in `.github/workflows/tests.yml`.

## Alternatives Considered

- **Leave tags floating** — rejected: a tag is a mutable pointer; a moved or compromised release can repoint it to inject code into CI with full access to workflow secrets.
- **Hand-pin SHAs** — rejected: a valid-but-wrong SHA (a real commit that is NOT the claimed `# vX`) still resolves and passes CI; only a tool that resolves each pin from the official upstream tag proves SHA-matches-tag.
- **Third-party-only pinning (exempt `actions/*`)** — rejected: first-party tags are still mutable pointers; the reduced maintenance overhead does not outweigh the defense-in-depth loss.

## Consequences

- Positive: A moved or compromised tag can no longer inject code into CI.
- Positive: Every action version is auditable at a glance (`# vX` comment) and reproducible (SHA).
- Positive: The `pinact-check` guard catches any new floating tag at PR time — the constraint cannot drift unnoticed.
- Negative: Slightly more Dependabot traffic — patch bumps within `@v7` now arrive as discrete SHA-bump PRs; these auto-merge under ADR-0017, so the cost is bounded.
- Negative: New contributors adding an action must run `pinact run` before pushing; the guard fails the gate on any miss with a clear diff.

## Lineage

Builds directly on **ADR-0017** (Dependabot full-CI + auto-merge makes SHA-bump cost negligible) and follows the **ADR-0070** drift-guard precedent (single-source a value, then enforce with a `--check` guard in CI).
