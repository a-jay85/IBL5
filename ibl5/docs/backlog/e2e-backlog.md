# E2E Test-Quality Backlog

Catalogue of test-quality reduction opportunities in the Playwright suite (`ibl5/tests/e2e`) and the
Vitest api-e2e suite (`ibl5/tests/api-e2e`). Generated by a parallel multi-agent audit (2026-06-28) over
all 110 specs + 3 api-e2e files (~18.5K LOC) + helpers/fixtures/config; cross-file dedup and every final
"useless / doesn't-prove-functionality" call made by hand (the top-3 D items verified against source).

**Companion to** [`maintenance-backlog.md`](maintenance-backlog.md) and [`a11y-backlog.md`](a11y-backlog.md);
same emoji-coded status taxonomy.

**Format** (detailed entries):

```
### <id> <short title>
**Location:** path:line
**Problem:** what's wrong
**Suggested direction:** sketch, not a plan
**Est. effort:** S / M / L
**Risk if untouched:** what bug/cost this enables
**Status:** glyph + note
```

Effort scale:
- **S** — single PR, < 1 day, one or few specs, no cross-cutting helper change
- **M** — multi-spec / new shared helper / parametrization sweep, 1–3 days
- **L** — suite-wide pattern shift, > 3 days

## Status taxonomy

**Status** — where the fix stands:
- ✅ **Implemented** — resolved on `master` (verified on disk).
- ◑ **Partial** — partially addressed; residual named in the note.
- 📋 **Planned** — a plan exists / PR open, not yet merged.
- ⬜ **Open** — no plan yet.
- 🚫 **Declined** — deliberately won't-do (rationale in the note).

**Automouse-readiness** (for items not ✅/🚫):
- 🟩 **Auto-mergeable** — test-only, behavior-preserving for production (green-green); arms auto-merge unattended.
- 🟦 **Automouse-safe, human-merge** — implementable unattended but a gate-14 trigger forces `auto_merge: false` (none here yet — test-only suite).
- 🟨 **Conditional** — would be 🟩 except it **may surface a real production defect** when the assertion is strengthened (the happy path must be confirmed to actually pass first; if it doesn't, a real bug exists → separate fix + human triage), OR it needs one upfront scope decision (shared helper shape).
- 🟥 **Not automouse-safe** — irreducible judgment.

> **Why so many 🟨 in Axis D:** strengthening a "doesn't-prove-functionality" test is green-green *only if the
> feature actually works*. Each D fix must first confirm the happy path passes the tightened assertion; a
> failure there is a genuine bug discovery, not a test regression. That branch needs a human.

> **Planning note:** 🟨 items set `auto_merge: false` (STOP-guard, human signoff) — the plan must confirm the
> happy path passes before treating a tightened assertion as green-green. Axis A, B, C, and E each package as
> one plan apiece (single coherent worktree per axis); Axis D is planned per-cluster instead (see below).
